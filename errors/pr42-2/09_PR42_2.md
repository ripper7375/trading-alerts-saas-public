The job failed due to TypeScript errors that indicate systemConfig and systemConfigHistory properties do not exist on the PrismaClient. Here’s the root cause and solution:

**Cause:**  
In your Prisma schema, SystemConfig and SystemConfigHistory are model names, not properties of the PrismaClient root. Accessing them via prisma.systemConfig and prisma.systemConfigHistory is correct only if you have generated the client after defining those models. If the client wasn’t regenerated or if there’s a mismatch between your code and the generated client, those properties will be missing and the error “Property 'systemConfig' does not exist on type 'PrismaClient'” will occur.

**How to fix:**

1. Ensure your Prisma schema includes the SystemConfig and SystemConfigHistory models, which it does ([schema.prisma, ref: b6932ab](https://github.com/ripper7375/trading-alerts-saas-public/blob/b6932abf38f80655e948a94f71eb52cdfb7fad97/prisma/schema.prisma#L773)).
2. Regenerate the Prisma client:

```sh
npx prisma generate
```

3. If you have renamed your Prisma client import or the generated client isn’t up-to-date, ensure you are using the correct import statement:

```typescript
import { prisma } from '@/lib/db/prisma';
```
